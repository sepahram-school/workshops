# Benchmark v2 Improvement Plan

## Problem Statement

The current benchmark has been reviewed by multiple senior data engineers who identified critical methodology flaws:

1. **Toy-scale**: 500 files × 310KB = 155MB total (should be 10GB+)
2. **Client bottleneck**: 2.1 MB/s means Python/boto3 is the bottleneck, not storage
3. **No MinIO baseline**: Can't evaluate alternatives without the original
4. **Unfair durability comparison**: RF=2 (3x copies) vs erasure coding (~1.5x)
5. **Single-host Docker**: Not testing distributed behavior (acknowledged limitation)
6. **No repeated runs**: Need 3+ runs with variance reporting
7. **No hardware specs**: Results are non-reproducible without them
8. **Small files anti-pattern**: 310KB is unrealistic; need 128MB+ files
9. **No multipart upload testing**: Large objects never exercised
10. **No resource monitoring**: CPU/RAM/disk during runs
11. **MinIO license claim wrong**: Server is still AGPLv3
12. **No feature parity matrix**
13. **RustFS ListObjectsV2 not properly diagnosed**
14. **No warmup phase**

## What We CAN Do on Single Machine

| Factor | v1 | v2 Target |
|--------|-----|-----------|
| Total data | 155 MB | ~10 GB |
| File sizes | 310 KB only | 1 MB / 32 MB / 128 MB |
| Systems tested | 3 | 4 (add MinIO) |
| Runs per config | 1 | 3 |
| Resource monitoring | None | docker stats |
| Variance reporting | None | mean ± stdev |

## What We CANNOT Fix (Honest Limitations)

- All "nodes" share one disk and NIC (loopback)
- No real network latency between nodes
- No true failure injection (Docker stop ≠ node failure)
- Not representative of multi-host production deployments
- Results are directional, not absolute

## Implementation Plan

### Step 1: Add MinIO Baseline

**Create** `docker-compose-minio.yml`:
- 3-node MinIO distributed cluster
- Port mapping: MinIO on host port **9200** (to avoid RustFS 9000 conflict)
- Credentials: `minioadmin` / `minioadmin123`
- Image: `minio/minio:latest`

**Update** `benchmark.py`:
- Add MinIO target config (endpoint `http://localhost:9200`)
- Add `minio` to `--target` choices

### Step 2: Pre-Generate Test Data

**Create** `generate-data.py`:
- Generate files in `data/{size}/` directories: `1mb/`, `32mb/`, `128mb/`
- 100 files per size tier = ~16GB total on disk
- Use `os.urandom()` for fast generation (no Parquet overhead for write benchmark)
- For read benchmark: also generate Parquet files with Polars (same schema as v1)
- Write to local disk, upload in separate benchmark phase
- This separates data generation time from upload measurement

**File sizes chosen:**
- 1 MB — tests S3 API overhead dominance
- 32 MB — typical Parquet partition size
- 128 MB — production Parquet file size, exercises multipart upload

### Step 3: Rewrite Benchmark Script

**Rewrite** `benchmark.py` with these changes:

**Architecture:**
- Read pre-generated files from `data/{size}/` (no generation during upload)
- Warmup phase: upload 5 files silently before measurement
- Per-run bucket naming: `bench-{target}-run{run_id}-{timestamp}`
- JSON output to `results/{target}/{size}/run-{n}.json`

**CLI changes:**
```
--target minio|seaweedfs|rustfs|librefs|all
--threads 10|20|50
--sizes 1mb,32mb,128mb
--runs 3
--files 100
--skip-read
```

**New features:**
- Warmup phase (5 files, excluded from metrics)
- Multiple runs with per-run JSON output
- Resource monitoring integration (subprocess calling `docker stats`)
- Per-run disk usage measurement after write phase
- Storage overhead ratio calculation

### Step 4: Add Resource Monitoring

**Create** `collect-metrics.ps1`:
- Runs `docker stats --no-stream` every 2 seconds
- Logs to CSV: timestamp, container, CPU%, mem usage, net I/O, block I/O
- Started as background job during benchmark, stopped after
- Aggregated into results JSON

**Add to benchmark script:**
- After write phase: measure disk usage on all data directories
- Calculate overhead ratio: `disk_used / logical_data_size`
- Include in results JSON

### Step 5: Create Orchestrator Script

**Create** `run-benchmark.ps1`:
- Collects hardware specs → `hardware-specs.md`
- Generates test data if not exists
- For each system (one at a time):
  1. Start cluster
  2. Start resource monitoring
  3. For each file size × 3 runs:
     - Run benchmark, save JSON
  4. Stop monitoring
  5. Stop cluster, cleanup
- Aggregate results → `results/summary.json`
- Generate report → `reports-v2.md`

### Step 6: Diagnose RustFS ListObjectsV2

Before running v2 benchmark, debug the RustFS issue:
- Test with `aws s3 ls` CLI directly against RustFS
- Test with `boto3` `list_objects_v2()` directly (outside DuckDB)
- Check RustFS container logs for error details
- Test with different prefix patterns
- File upstream issue if confirmed bug
- If unfixable: exclude RustFS from read comparisons, note in report

### Step 7: Fix Report Content

**Rewrite** `reports.md`:

**Fix errors:**
- MinIO license: Server is **still AGPLv3** (features moved to proprietary, not the server)
- RustFS: Describe as "observed in our setup" not "known bug"
- Soften conclusions to match evidence strength

**Add sections:**
- Hardware Specifications (CPU, RAM, disk type, Docker version)
- Methodology Limitations (honest about what single-machine can/cannot test)
- Storage Overhead Analysis (3x vs 1.5x, effective throughput per TB)
- Feature Parity Matrix (S3 features: listing, versioning, encryption, IAM, lifecycle, etc.)
- Operational Comparison (startup time, memory usage, monitoring, upgrade path)
- Resource Utilization Charts (CPU/RAM during benchmark)

**Restructure conclusion:**
- "SeaweedFS showed favorable performance in our single-machine microbenchmark"
- "Broader evaluation on production hardware is required before migration"
- Remove "Production-Grade" from title

### Step 8: Document Hardware Specs

**Create** `hardware-specs.md`:
- OS, CPU, RAM, disk type, Docker version
- Docker Desktop configuration (WSL2, memory limits)
- Network (loopback only)
