# Plan: Standardize Storage Configuration Across Systems

## Current State

| System | Nodes | Data Protection | Overhead | Configurable? |
|--------|-------|----------------|----------|---------------|
| **MinIO** | 3 | Erasure coding (distributed mode) | ~1.5x | No — automatic in distributed mode |
| **RustFS** | 3 | Erasure coding (MinIO-compatible) | ~1.5x | No — automatic in distributed mode |
| **libreFS** | 3 | Erasure coding (MinIO fork) | ~1.5x | No — automatic in distributed mode |
| **SeaweedFS** | 3 volumes | Replication (`defaultReplication=001`) | ~1x (RF=1) | Yes — `001`=RF=1, `002`=RF=2, `003`=RF=3 |

## Analysis

### MinIO / RustFS / libreFS — Already Consistent

All three use **erasure coding** in 3-node distributed mode. They are already equivalent:
- 3 nodes, each with 1 disk
- Erasure coding splits data across nodes with parity
- ~1.5x storage overhead (1.5 TB physical per 1 TB usable)
- **No explicit config needed** — erasure coding is automatic in distributed mode

**No changes needed.**

### SeaweedFS — Different Architecture

SeaweedFS uses **replication**, not erasure coding. These are fundamentally different:

| | Erasure Coding (MinIO/RustFS/libreFS) | Replication (SeaweedFS) |
|--|---------------------------------------|-------------------------|
| How it works | Data split into chunks + parity bits across nodes | Full copies on each node |
| 3-node durability | Tolerates 1 node failure | RF=2 tolerates 1 node failure |
| Storage overhead | ~1.5x | RF=1: ~1x (1 copy/node), RF=2: ~3x (3 copies) |
| CPU cost | Higher (encoding/decoding) | Lower (just copying) |

### Can We Equalize?

**No — SeaweedFS fundamentally cannot use erasure coding.** It's a different architecture.

The closest equivalent would be to set SeaweedFS to **RF=2** (`defaultReplication=002`), which provides the same fault tolerance as erasure coding (tolerates 1 node failure). But:
- **Overhead would be ~3x** (3 copies vs 1.5x for erasure coding)
- **CPU overhead would be lower** (no encoding/decoding)
- **Write throughput would decrease** (more data to write)

## Recommendation

**Keep the current configurations.** They are already fair:

1. **MinIO / RustFS / libreFS**: 3-node distributed, erasure coding — already identical
2. **SeaweedFS**: RF=1 (`defaultReplication=001`) — this is the **production default** and represents how SeaweedFS is actually deployed

Changing SeaweedFS to RF=2 would:
- Make it slower (more writes per object)
- Use more disk (3x vs 1.5x)
- Not make the benchmark "more fair" — it would just penalize SeaweedFS

**The fair comparison is: each system at its production-recommended default.**

## Decision

**Keep current defaults.** Each system at its production-recommended default. No changes needed to compose files or benchmarks.
