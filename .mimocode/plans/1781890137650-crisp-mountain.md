# Plan: Replace SQLite with Turso in DuckLake Benchmark

## Problem
DuckLake heavy mode with SQLite measures **SQLite WAL contention** (13% write success), NOT S3 storage performance. The benchmark doesn't actually test the storage systems under concurrent lakehouse workloads.

## Solution
Use **Turso (pyturso)** — a drop-in SQLite replacement with MVCC (concurrent writes) — as DuckLake's metadata backend. This removes the SQLite bottleneck so the benchmark measures S3 throughput.

## Key Question: Does DuckLake Support Turso?

DuckLake uses `ATTACH 'ducklake:sqlite:metadata.db'` — this calls DuckDB's internal SQLite scanner, NOT Python's sqlite3 module. Turso is a Python-level sqlite3 replacement. So **DuckLake cannot directly use Turso** as its metadata backend.

## Workaround: Two Approaches

### Approach A: Separate metadata DBs per writer (Simple, Already Tested)
- Each writer thread gets its own DuckLake metadata DB
- No SQLite contention — each writer is independent
- Tests: S3 throughput under concurrent independent writes
- Concurrency limitation: SQLite WAL is removed, but DuckLake metadata overhead per-write remains

### Approach B: Use Turso as standalone coordination layer
- Replace DuckLake's metadata with direct Turso connections
- Writers coordinate via Turso instead of SQLite
- More complex, but tests Turso + S3 together

**Recommendation: Approach A** — simpler, already proven to work, directly measures S3 performance.

## Changes Required

### benchmark_ducklake.py

1. **`run_heavy_phase`**: Each writer creates its OWN metadata DB (not shared)
   - Current: `shared_meta = f"metadata_heavy_shared_{run_id}.db"` (shared)
   - New: `meta_file = f"metadata_heavy_{run_id}_w{worker_id}.db"` (per-worker)
   
2. **`run_concurrent_phase`**: Same — per-worker metadata DBs
   
3. **Remove `meta_files_lock`** — no longer needed since each worker has its own DB

4. **Add Turso as optional metadata backend** (future):
   ```python
   if use_turso:
       import turso
       conn = turso.connect(f"metadata_{worker_id}.db")
   ```
